# Submit plan: splitting the visualizer prototype into reviewable pieces

## Purpose

PRs #6–#12 were written quickly as exploratory prototyping. Changes were
hacked together, some were later changed or abandoned, and several landed in a
different PR than the topic they really belong to. This document reorganizes
all of that work into **independent tracks** that can be reviewed and submitted
as small, coherent units, and gives a recommended ordering.

It is grounded in a file-by-file reading of the actual branch diffs (not just
the PR descriptions). Where a change sits in the "wrong" PR, that is called out
explicitly in [Regrouping](#regrouping-changes-that-are-in-the-wrong-pr).

content is the visualizer UI, which is split into pieces below.

## The tracks (independently-ownable components)

Six tracks. The first five are infrastructure that can land on master in
parallel (with one stacking dependency, C→B). The sixth is the visualizer,
which consumes all of them and is itself split into ordered sub-pieces.

```
  A  Linear Resolved AST DebugString          (from #10)            ─┐
  B  SQLBuilder: pipe framework + 1:1 ops      (from #6)            ─┤
  C  SQLBuilder: readable names  (stacks on B) (from #9)            ─┤  land on
  D  AST-node resolution info     (from #8 + #7's extension)        ─┤  master,
  E  Declarative formatters (box/html)         (from #7)            ─┘  mostly
                                                                        parallel
  F  execute_query visualizer  (from #7-integration, #11, #12)  ← consumes A–E
```

### Track A — Linear Resolved AST DebugString rendering

**Source:** PR #10, entirely. **Independence:** total — touches only
`resolved_ast/` and `tools/execute_query/`, depends only on master.

- `DebugStringConfig::linear_mode` (+ `omit_pipe_input_scan_field`) on
  `resolved_node.h`; renderer changes in `resolved_node.cc` (`DebugStringImpl`
  refactored to extract `DebugStringBody`, lays out the `|>` chain with a faint
  left-edge connector).
- New virtual `ResolvedScan::GetPipeInputScan()`, auto-generated by
  `gen_resolved_ast.py` (new `is_pipe_input_scan` field marker + name inference
  for `input_scan`/`scan`); hand-written overrides for `ResolvedJoinScan`,
  `ResolvedWithScan`, `ResolvedSetOperationScan`, `ResolvedRecursiveScan`.
- execute_query flags `--linear_resolved_ast`, `--linear_and_tree_resolved_ast`.

**Review notes:** small and honest (9 files, +334/−28); **no generated
`*_generated.*` files are committed** — the overrides materialize at build
time. The four commits are feature → unit test → two rendering refinements
(vertical bar → faint dotted connector); the refinements only re-baseline the
one golden, so review as a **single logical unit** (final state is what
matters).

**Why it must land early:** the visualizer's Resolved AST pane and the
`query_graph` builder both depend on `linear_mode` and `GetPipeInputScan()`.

### Track B — SQLBuilder pipe framework + 1:1 operator completeness

**Source:** PR #6, entirely (plus one completeness item misfiled in #9 — see
[Regrouping](#regrouping-changes-that-are-in-the-wrong-pr)).
**Independence:** self-contained on master; affects pipe-target output only,
standard SQL output unchanged.

This is the "make pipe-syntax generation actually work for every scan kind"
track. Two halves:

1. **The append-as-you-go framework.** New methods in `sql_builder.{cc,h}`:
   `GetInputPipeSQL`, `MakePipeQueryExpression`, `AppendPipeOperator`,
   `RenderPipeOperatorInput`, `FinishPipeOperatorScan`,
   `GetPipeSubpipelineListSQL`. Mode scoping via `ScopedTargetSyntaxMode` /
   `target_syntax_mode_override_` / `CurrentTargetSyntaxMode()`. Operator-chain
   support in `query_expression.{cc,h}` (`MarkAsPipeOperatorChain`,
   `IsPipeOperatorChain`, `GetFromClausePipeSQL`).
2. **1:1 pipe-operator emission for every scan kind.** Converted `Visit*`
   methods: `FilterScan`→`|> WHERE`, `OrderByScan`→`|> ORDER BY`,
   `JoinScan`/`ArrayScan`→`|> JOIN [UNNEST]`, `SetOperationScan`→`|> UNION/…`,
   `LimitOffsetScan`→`|> LIMIT`, `SampleScan`→`|> TABLESAMPLE`,
   `AssertScan`/`LogScan`/`PipeIfScan`, and the previously-`RET_CHECK_FAIL`
   `PipeForkScan`/`PipeTeeScan`/`GeneralizedQueryStmt`. Subpipeline rendering
   rewritten (`VisitResolvedSubpipeline(InputScan)` now uses the empty-chain
   model instead of the old `UNNEST(__SubpipelineInput__)` hack).

**Suggested internal ordering** (each builds on the last): (1) mode scoping +
operator-chain `QueryExpression`; (2) framework primitives; (3) `FilterScan` +
`OrderByScan`; (4) subpipeline rendering + `Assert`/`Log`/`If`; (5)
`Fork`/`Tee`/`GeneralizedQueryStmt`; (6) `Join`/`Array`/`SetOp`/`Limit`/
`Sample`; (7) aggregate-in-pipe correctness. These can be one PR or a short
stack.

**Review notes:** large golden churn (`sql_builder` `.test` files) is
mechanical — review should focus on `sql_builder.cc` / `query_expression.cc`.

### Track C — SQLBuilder readable column names (stacks on B)

**Source:** PR #9, the naming/readability subset only. **Stacks on B.**

Four readability strands, mostly independent, **except one indivisible bundle**:

- **Omit redundant `AS`** (`query_expression.cc`): `AliasIsRedundant` +
  `omit_redundant_aliases`. Self-contained.
- **Inline simple `GROUP BY`** (`query_expression.cc`): `TryAppendGroupByClause`
  inline path, `ComputePipeGroupByForcedExtendColumns`. Depends on the redundant-
  `AS` change.
- **⊗ Indivisible bundle — readable names + leaf-wrap removal + global alias
  uniqueness.** `GetColumnAlias` (derive alias from `ResolvedColumn` name, not
  `a_N`), `MakeUniqueColumnAlias` / `ColumnAliasIsTaken` (**case-insensitive**),
  `TryUseLeafTableScanColumns` (expose leaf-table columns by natural name, drop
  the `|> SELECT … |> AS` wrapper) and its alias-reservation loop, the
  `GetSelectList` unique-key materialization, and the **deletion** of
  `AppendPipeAggregateOutputProjectionIfNeeded`. **These cannot be subdivided:**
  removing the leaf wrapper puts leaf columns into the same namespace as derived
  aliases, so without global (case-insensitive) uniqueness the output is
  ambiguous/invalid SQL. The #9 commit log already flagged this ("reveals
  uniqueness entanglement").
- **Docs:** `newdocs/pipe-sql-builder.md` — land last so it describes final
  behavior.

**Review notes:** this is the bulk of #9's 70+ golden-file churn. It is a
**quality** dependency of the visualizer's SQLBuilder pane (output is readable),
not a compile dependency — F can technically build on B alone, but lands ugly.
Recommend C before the visualizer's text panes (F2).

### Track D — AST-node resolution-info capture (unify #8 + #7's extension)

**Source:** PR #8 **and** the resolver/analyzer half of PR #7 — these are the
same feature split across two PRs. #7's `ast_node_resolved_info.h` is a **strict
additive superset** of #8's (verified byte-for-byte); the plumbing
(`resolver.h` member, `analyzer_output.{h,cc}`, `analyzer.cc`,
`analyzer_impl.cc`, BUILD) is **identical** between them. **Land as one unified
track, not #8-then-re-touch.**

- **Struct** (`public/ast_node_resolved_info.h`): from #8 — `TableScanInfo`,
  `ResolvedScanInfo` (is_pipe_operator + input/output NameLists),
  `ASTNodeResolvedInfoMap`. From #7's extension — `FunctionCallInfo`
  (signature), `StatementInfo` (output columns / value-table), and the
  `node_title` field.
- **Resolver wiring:** from #8 — `MakeScanForTable` (table_scan_info),
  `ResolvePipeOperatorList` (resolved_scan_info). From #7 — `ResolveQuery` /
  `ResolveFromQuery` (resolved_scan_info + node_title; new
  `ResolveQueryOptions::node_title_hint`), `resolver_expr.cc`
  (function_call_info), `resolver_stmt.cc` (statement_info), and `node_title`
  on all of them.
- Threaded resolver → `AnalyzerOutput::ast_node_resolved_info_map()`.

**Independence:** self-contained on master; non-breaking (default empty map).
The map keys are raw `const ASTNode*` (no numeric IDs / cross-refs in the
struct — correspondence is by pointer). Consumed only by the visualizer.

### Track E — Declarative formatters (box / HTML)

**Source:** the parser-library half of PR #7. **Independence:** parser-only;
no analyzer/Resolved-AST dependency.

- `parser/rewritable_string.{h,cc}` — offset-keyed range-wrapping utility (no
  project deps beyond base/absl). Lowest layer.
- `parser/html_formatter.{h,cc}` — `SqlToHtml`: wraps each AST node's source
  range in `<div class="ast …">`, preserving original text. **Depends on
  `rewritable_string`** + `:parser`. Independent of box_formatter.
- `parser/box_formatter.{h,cc}` — `SqlToBoxHtml`: Wadler/Oppen width-aware
  pretty-printer that re-lays-out the SQL. Depends only on `:parser`. **Stays
  analyzer-free by design:** its only hook to resolver data is the optional
  `BoxAnnotator = std::function<std::string(const ASTNode*)>` callback (the
  caller supplies pre-rendered HTML), so the coupling to resolution-info lives
  in execute_query, not here.

**Note:** there is no shared `EscapeHtml` — each formatter carries its own
file-local copy (a deliberate decoupling). The `FormatResultTableDiffHtml`
helper added to `common/reflection_helper` is **not** a formatter; it renders
NameList diffs for the visualizer hover boxes and belongs with **Track F**, not
here.

**Could the visualizer skip the box formatter?** No for the Input pane (it needs
`SqlToBoxHtml` + `BoxAnnotator` to emit the `.rect` divs that carry per-node
hover info), but **yes** for the SQLBuilder pane — that pane uses the
pre-existing `LenientFormatSql` plus a local `|>`-splitter
(`SegmentPipeSqlToHtml`), not box_formatter. `html_formatter` is used by the
parse-mode "Formatted SQL" tab (part of F1), not by the three visualizer panes.

### Track F — execute_query visualizer

**Source:** the execute_query integration in PR #7 (analyze-mode viewer),
PR #11 (phase 1, multi-column text), PR #12 (phase 2, graph). **Consumes
A, B, C, D, E.** Split smallest-reviewable-first:

- **F1 — Formatters wired into analyze mode.** The execute_query half of #7:
  `formatted_sql_html` / `formatted_analyzed_html` writer hooks,
  `ResolvedInfoHoverHtml`, the Default/Simple-box/Full-box format tabs, and the
  original 140-line info-panel `query_viewer.js`. *Needs E + D.* No Visualize
  mode yet — reviewable alone.
- **F2 — `kVisualize` tool mode + 3-pane text shell.** `ToolMode::kVisualize`
  (introduced in #11, **not** #7), `VisualizeQuery()` (re-analyze with
  rewriters off + parse locations on; emit Input SQL / Resolved-AST-linear /
  SQLBuilder-pipe), `VisualizationData`, the `visualized()` writer (text + web),
  the `.viz` three-column resizable HTML/CSS + linked-state JS — panes as plain
  text first. *Needs A (AST pane) + B (SQLBuilder pane) + F1.*
- **F3 — Input pane node-info + details box.** `RenderBoxHtmlWithNodeInfo`, the
  `.viz-info` details box, NameList/signature rendering
  (`FormatResultTableDiffHtml`). *Needs D + F2.*
- **F4 — Cross-pane correspondence.** `scan_ids` from `DebugStringHtml`,
  `data-node-id`/`data-corresp` tagging across the three panes, the SQLBuilder
  `pipe_operator_markers()` side-channel + `SegmentPipeSqlToHtml`, the JS
  one-hop highlight model. *Needs A + the SQLBuilder marker hook + F3.*
- **F5 — Post-rewrite section.** The diff-and-show-if-changed read-only AST
  block. Small. *Needs F2.*
- **F6 — Graph / tree view** (≈ today's #12). `query_graph.{cc,h}` + test (its
  own `cc_library`, deps = `resolved_ast` + absl only; walks `GetPipeInputScan`
  spine, emits `r<n>` node ids matching the panes), `resolved_graph_json`
  plumbing, the per-column view-mode `<select>`, the embedded-JSON + inline-SVG
  graph renderer (~270 JS lines; self-contained, no elkjs/d3), view-aware
  correspondence + scroll-into-view. *Needs A + F4.*

**One prerequisite that lives in the SQLBuilder, not the UI:** the
`record_pipe_operator_markers` / `pipe_operator_markers()` opt-in recording mode
on `SQLBuilder` (maps each emitted `|>` back to its producing `ResolvedScan`).
It does not exist in base and is needed by F4. It is a small SQLBuilder-side
addition — land it with Track B (or as a tiny follow-on to B) so F4 has it.

## Regrouping: changes that are in the wrong PR

These are the misfiled / tangled changes to fix while re-cutting:

1. **Resolution-info is split across #8 and #7.** #8 has the base struct +
   wiring; #7 silently carries the extension (`FunctionCallInfo`,
   `StatementInfo`, `node_title`, and the `resolver_expr.cc` / `resolver_stmt.cc`
   instrumentation). → **Merge both into Track D** and land once.

2. **#7 bundles three unrelated things.** Declarative formatters (→ E), the
   resolution-info extension (→ D), and the execute_query analyze-mode
   integration (→ F1). → **Split #7 along those three seams.**

3. **A completeness fix is misfiled in the naming PR.** The degenerate
   top-level `ProjectScan` 1:1 change (drop `IsDegenerateProjectScan` elision in
   pipe mode so every scan still emits its `|> SELECT`) is in #9 but is a
   *completeness* item, not naming. → **Move it to Track B.**

4. **`FormatResultTableDiffHtml` is in the wrong layer.** It lives in
   `common/reflection_helper` (#7) but is purely a visualizer hover-box
   renderer. → **Conceptually belongs with Track F3**; it can stay physically in
   `reflection_helper` (no compile coupling) but should be reviewed as part of
   F, not E.

5. **#11/#12 re-import upstream as squashed commits.** If A–E land
   independently on master, **#11 and #12 must be rebased** to drop the squashed
   re-imports and keep only their novel visualizer commits (the F-pieces);
   otherwise they duplicate history.

## Recommended submission order

Infrastructure first (A, B, D, E can go in parallel; C stacks on B), then the
visualizer pieces in dependency order.

| Step | Track / piece | Depends on | Rough review size |
|------|---------------|-----------|-------------------|
| 1 | **A** Linear Resolved AST | master | small (9 files) |
| 2 | **B** SQLBuilder pipe framework + 1:1 ops (+ marker hook) | master | large (goldens) |
| 3 | **D** Resolution-info capture (unified #8+#7) | master | small/medium |
| 4 | **E** Declarative formatters (box/html/rewritable_string) | master | medium |
| 5 | **C** SQLBuilder readable names | B | large (goldens) |
| 6 | **F1** Formatters in analyze mode | D, E | medium |
| 7 | **F2** `kVisualize` + 3-pane text shell | A, B, F1 | medium |
| 8 | **F3** Input node-info + details box | D, F2 | medium |
| 9 | **F4** Cross-pane correspondence | A, B-marker, F3 | medium |
| 10 | **F5** Post-rewrite section | F2 | small |
| 11 | **F6** Graph / tree view (`query_graph`) | A, F4 | medium |

Steps 1–4 are unblocked immediately and independent of each other. C (step 5)
should precede the visualizer's text panes so the SQLBuilder pane reads well.
Within F, each piece is shippable on its own and leaves the tool in a working
state.

  sets still use the EXTEND form and can still produce ambiguous aliases, and
  the `aggregatescan_1` range-variable name is still ugly. Decide whether Track
  C closes these or explicitly scopes them out.
- **Golden regeneration.** B and C both require regenerating `sql_builder`
  goldens (and #9 touched 70+); make sure each PR's goldens are regenerated
  against *that* PR's state, not the bundled end-state.
- **The SQLBuilder marker side-channel** (`pipe_operator_markers`) — confirm its
  current home in the branch history and attach it to Track B so F4 isn't
  blocked.
