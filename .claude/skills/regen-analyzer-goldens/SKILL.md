# Regenerating analyzer `.test` goldens

The analyzer's golden tests live in `googlesql/analyzer/testdata/*.test` (directives +
SQL + expected resolved-AST / `[SQLBUILDER_OUTPUT]` blocks). There is **no `update
goldens` command** in the open-source export — `extract_test_output.py` is not shipped.
Instead the test driver writes the regenerated golden *into the test log*, and you
extract it from there with the bundled `extract_golden.py`.

## 1. Find the target for a `.test` file

Each `testdata/<name>.test` is wired to `//googlesql/analyzer:analyzer_<name>_test`
(generated by `gen_analyzer_test` in `googlesql/analyzer/builddefs.bzl`,
`name = "analyzer_" + filename.replace(".","_")`). Examples:
`pipe_join.test` → `analyzer_pipe_join_test`, `set_operation.test` → `analyzer_set_operation_test`.

To find every test affected by a **SQLBuilder** change (77 files record SQLBuilder output):
```bash
for f in $(grep -rl 'show_sqlbuilder_output' googlesql/analyzer/testdata/ | sed 's#.*/##; s/\.test$//'); do
  echo "//googlesql/analyzer:analyzer_${f}_test"
done
```

## 2. Run the test (the "failure" is expected — it's how the new golden gets logged)

The driver regenerates output by default
(`--file_based_test_driver_generate_test_output` defaults to `true`). The flags below are
the ones that work on this dev box (see `CLAUDE.local.md`); adjust for your environment:
```bash
bazel test --config=clang --jobs=2 --nobuild_runfile_links --dynamic_mode=off \
  --test_output=errors --nocache_test_results \
  //googlesql/analyzer:analyzer_<name>_test
```

## 3. Triage BEFORE rewriting the golden — is the diff benign?

Not every diff means "update the golden." Grep the log for the signature:

| Log signature | Meaning | Action |
|---|---|---|
| `BEGIN TEST DIFF` only, no error | Benign golden churn (only recorded text changed; SQL still valid) | Safe to regenerate |
| `ERROR while analyzing SQLBuilder output: ...` | **Real bug** — SQLBuilder emitted invalid/non-equivalent SQL | Fix the code, do **not** regenerate |
| `Pipe-operator markers changed the generated SQL structure` | **Real bug** — `/*S<n>*/` visualizer markers altered output | Fix the code, do **not** regenerate |

```bash
LOG=bazel-testlogs/googlesql/analyzer/analyzer_<name>_test/test.log
grep -E 'ERROR while analyzing SQLBuilder output|markers changed the generated' "$LOG" || echo "no real errors — golden churn only"
```

**Also check the baseline.** Several testdata files fail *pre-existing*, unrelated to your
change — notably `set_operation` (`TestInvalidTypeAnnotations` proto-catalog error),
`pipe_with` (`Expected ")" but got keyword CROSS`), and `pipe_tablesample`. If a file fails
and has no `[SQLBUILDER_OUTPUT]` blocks (or the failing hunk doesn't touch your feature),
it was already red. Confirm with `git stash push -- <changed files>` and re-run at baseline.

## 4. Extract the regenerated golden from the log

The log embeds the new file between `****TEST_OUTPUT_BEGIN****` / `****TEST_OUTPUT_END****`
markers (first block tagged `NEW_TEST_RUN <path>`), split across blocks with
`***MERGE_TOO_LONG_LINE***` sentinels for long lines. **Do not use `patch`** — the driver
numbers hunks per-test-case, so `@@` line numbers don't match the real file and patches
fail/fuzz. Use the bundled extractor (it reassembles + rejoins split lines):

```bash
python3 .claude/skills/regen-analyzer-goldens/extract_golden.py \
  bazel-testlogs/googlesql/analyzer/analyzer_<name>_test/test.log \
  googlesql/analyzer/testdata/<name>.test
# add --dry-run first to preview line count without writing
```

The `test.outputs/` undeclared-outputs dir is **empty** — the golden is only in `test.log`.

## 5. Verify

```bash
git diff googlesql/analyzer/testdata/<name>.test    # only expected blocks should change
bazel test --config=clang --jobs=2 --nobuild_runfile_links --dynamic_mode=off \
  //googlesql/analyzer:analyzer_<name>_test          # should now pass
```
Sanity-check the diff touches only the blocks your change should affect (e.g. only
`[SQLBUILDER_OUTPUT]` / `[SQLBUILDER_PIPE]`, not resolved-AST structure).

## Many files at once

Loop steps 2 + 4 over the target list from step 1. A fast "what's still broken vs. just
churn" sweep:
```bash
bazel test ... $TARGETS --test_output=errors --keep_going 2>&1 \
 | grep -E 'ERROR while analyzing SQLBuilder output|markers changed the generated' \
 | sed -E 's/.*output: [A-Z_]+: //; s/\[at [0-9:]+\]//' | sort | uniq -c | sort -rn
```
Empty output = only golden diffs remain → safe to regenerate all of them.
